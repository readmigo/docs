# Readmigo Audio: 公版精品有声书

> 为公版经典打造精品有声书——重制已有高质量录音，原创制作缺失的部分

## 一句话定义

**Readmigo Audio** = 公版书的精品有声书制作者。对已有高质量真人录音（LibriVox 等）进行深度重制，对缺失录音的书籍进行 AI+真人原创制作，最终为每一本 SE 精校文本配上高质量音频。

### 为什么是两条业务线

| 维度 | 业务线 1: Readmigo App | 业务线 2: Readmigo Audio Lab |
|------|----------------------|----------------------------|
| 角色 | 消费者（第一方客户） | 生产者（内容供应商） |
| 产出 | 用户体验、学习功能 | 精品有声书数据集 |
| 受众 | 语言学习者 | 所有需要公版有声书的平台 |
| 壁垒 | 三位一体体验（文本+音频+翻译） | 制作质量与覆盖规模 |
| 开放性 | Readmigo 专属 | 开放数据，类似 SE 的开源模式 |

**关键洞察**：SE 的成功在于它是一个**独立的内容品牌**，不依赖单一 App。Readmigo Audio Lab 同理——精品有声书本身就是有价值的产出，Readmigo App 只是第一个也是最好的消费者。

## 市场空白

### 文本领域（已被填补）

```
Project Gutenberg (原始文本, 1971, 55年)
        ↓  精校、排版、元数据、封面设计
Standard Ebooks (精品文本, 2014, ~9年)
        ↓  语言学习增强（翻译、标注、分级）
Readmigo (学习型精品文本)
```

### 音频领域（空白）

```
LibriVox (志愿者录音, 2005, 21年, 质量参差不齐)
        ↓
        ↓  没有人做精品制作
        ↓
        ↓  ← Readmigo Audio 要填补的空白
        ↓
Readmigo Audio (精品有声书)
        ├── 轨道 A: 重制 LibriVox 高质量录音
        └── 轨道 B: 原创制作（AI TTS + 真人）
```

### 竞品现状

| 竞品 | 定位 | 是否做了精品制作 |
|------|------|-----------------|
| LibriVox | 志愿者朗读，接受所有"可理解"录音 | 否，类似 PG 的"来者不拒" |
| Loyal Books | LibriVox 聚合展示 | 否，纯搬运 |
| Lex Books | 现代播放器 + LibriVox 音频链接电子书 | 否，UI 包装，音频未处理 |
| Digitalbook.io | PG + LibriVox 聚合搜索 | 否，纯索引 |
| PG + Microsoft/MIT | AI TTS 生成 ~5000 本有声书 | 否，机器合成，无策展 |
| Standard Ebooks | 精品电子书 | 不涉及音频 |

**结论：公版有声书的精品制作者不存在，这是一个真实的市场空白。**

## 产品愿景

### 双轨制：重制 + 原创

```
对每一本 SE 书籍：

LibriVox 有高质量录音？
  │
  ├── 是 → 轨道 A: 重制
  │         下载 → 筛选最佳版本 → 音频标准化 → 章节对齐 → 审核
  │
  └── 否 → 轨道 B: 原创
            SE 文本 → AI TTS 生成 / 真人录制 → 后期处理 → 审核
            （天然与 SE 章节对齐，无匹配问题）

两条轨道 → 统一品质标准 → 精品有声书
```

### 轨道 A: 重制（基于 LibriVox）

| 步骤 | 说明 |
|------|------|
| 匹配 | LibriVox title + author → Readmigo Book ID |
| 择优 | 同书多版本，选最佳（Solo > Group、高码率、新录音） |
| 标准化 | 音量归一化（-16 LUFS）、格式统一、静音裁剪 |
| 章节对齐 | 音频章节 ↔ SE 文本章节映射（模糊匹配 + 人工审核） |
| 元数据 | 朗读者档案、时长、音质评级 |

### 轨道 B: 原创（AI + 真人）

| 步骤 | 说明 |
|------|------|
| 文本准备 | SE 精校文本按章节拆分，去除 HTML 标签 |
| 音频生成 | 高质量 AI TTS（OpenAI TTS / ElevenLabs）或招募真人朗读 |
| 后期处理 | 音量归一化、格式统一（与轨道 A 相同标准） |
| 章节对齐 | 天然对齐（从 SE 文本直接生成，无匹配问题） |
| 元数据 | 标注来源（AI 生成 / 真人录制）、朗读者/模型信息 |

### 轨道 B 的优势

| 对比维度 | 轨道 A（重制 LibriVox） | 轨道 B（原创制作） |
|---------|----------------------|-------------------|
| 质量上限 | 受限于原始录音 | 完全可控 |
| 章节对齐 | 需要复杂匹配算法 | 天然对齐，零匹配成本 |
| 覆盖率 | 受限于 LibriVox 目录 | 任何有 SE 文本的书都可制作 |
| 成本 | 低（加工成本） | 中（AI TTS API 费用或真人录制费用） |
| 壁垒 | 低（他人也可重制 LibriVox） | 高（原创内容是独家壁垒） |

## 核心差异化：三位一体

```
┌─────────────────────────────────────────────┐
│              Readmigo 独有价值               │
│                                             │
│   ┌───────────┐                             │
│   │  SE 精校   │ ← 高质量文本               │
│   │   文本     │                             │
│   └─────┬─────┘                             │
│         │                                   │
│         ▼                                   │
│   ┌───────────┐    ┌───────────┐            │
│   │ 精品音频   │───→│  章节/段落 │            │
│   │ (重制+原创)│    │   同步    │            │
│   └─────┬─────┘    └───────────┘            │
│         │                                   │
│         ▼                                   │
│   ┌───────────┐                             │
│   │ 10语言翻译 │ ← 多语言支持               │
│   └───────────┘                             │
│                                             │
│   文本 + 音频 + 翻译 = 沉浸式语言学习       │
└─────────────────────────────────────────────┘
```

没有任何现有平台同时拥有这三者：
1. **SE 级精校文本** —— Lex Books 用的是 PG 原始文本
2. **精品制作的音频（重制+原创）** —— 所有平台都是 LibriVox 原始数据
3. **多语言翻译** —— 没有竞品提供

## 数据架构

### 存储结构

```
R2 Storage:
  books/{bookId}/
    ├── cover.jpg              ← 已有
    ├── chapters/{cid}.html    ← 已有
    ├── translations/          ← 已有
    └── audio/
        ├── metadata.json      ← 音频元数据（来源、朗读者、轨道类型）
        └── chapters/
            ├── 01.mp3         ← 精品音频文件
            ├── 02.mp3
            └── ...
```

### 数据模型关系

```
Book (SE 精校文本)
  ├── Chapter[] (文本章节)
  │     └── ChapterTranslation[] (翻译)
  │
  └── Audiobook (精品有声书)
        ├── source: LIBRIVOX_REMASTER | AI_TTS | HUMAN_RECORDING
        └── AudiobookChapter[]
              ├── audioUrl (R2 音频 URL)
              ├── bookChapterId → Chapter (文本章节关联)
              └── timestamps[] (段落同步数据)
```

## 研发路线图

### 总览

```
Phase 1          Phase 2              Phase 3         Phase 4         Phase 5
客户端 TTS  →  Audio Lab 管道  →  精品有声书制作  →  导入+播放  →  章节级同步
                (双轨制)           & 导入

 能听"          生产能力"          内容"              精品"           沉浸体验"
```

**逻辑**：TTS 先让所有书"能听"（验证需求 + 收集数据）→ 建设双轨制生产管道 → 产出精品内容逐步替换 TTS → 深度同步实现终极体验

---

### Phase 1: 客户端 TTS（让用户先"能听"）

**目标**：所有书籍立刻拥有语音朗读能力，零服务端成本

| 维度 | 说明 |
|------|------|
| 实现方式 | iOS `AVSpeechSynthesizer` / Android `TextToSpeech` |
| 服务端改动 | 无（客户端直接用现有章节文本内容） |
| 覆盖率 | 100%（任何有文本的书都能朗读） |
| 成本 | 零 |
| 音质 | 中等（系统 TTS，非精品） |

**价值**：
- 立刻验证"用户是否想听书"这一核心假设
- 收集用户行为数据（哪些书被听得最多 → 优先制作精品音频）
- 为 Phase 2 提供选书优先级排序依据

**交付物**：App 内任意书籍可 TTS 朗读

---

### Phase 2: Audio Lab 双轨制管道建设（生产能力）

**目标**：搭建重制+原创双轨制精品有声书生产管道

#### 轨道 A 管道（重制 LibriVox）

| 任务 | 说明 |
|------|------|
| LibriVox API 目录抓取 | 获取 20,000+ 有声书目录与元数据 |
| 自动匹配算法 | LibriVox title + author → Readmigo Book ID |
| 覆盖率摸底报告 | 有多少 SE 书能匹配到 LibriVox 高质量录音 |
| 多版本择优筛选 | 同一本书多个录音版本，选最佳 |
| 音频标准化工具链 | 音量归一化、格式统一、静音裁剪 |
| 章节对齐算法 | 音频章节 ↔ SE 文本章节映射 |

#### 轨道 B 管道（原创制作）

| 任务 | 说明 |
|------|------|
| 文本预处理工具 | SE HTML → 纯文本，按章节拆分 |
| AI TTS 集成 | 接入 OpenAI TTS / ElevenLabs API |
| 音频后期处理 | 与轨道 A 相同的标准化流程 |
| 质量评审流程 | AI 生成音频的人工抽检 |

#### 决策矩阵

```
对每一本 SE 书籍：

LibriVox 有匹配？ ──→ 录音质量达标？ ──→ 轨道 A（重制）
     │                    │
     │ 否                 │ 否
     ▼                    ▼
轨道 B（原创制作）    轨道 B（原创制作）
```

**交付物**：双轨制可复用的精品有声书生产管道 + 覆盖率报告

---

### Phase 3: 精品有声书制作与导入

**目标**：首批精品有声书通过管道产出并导入系统

| 任务 | 说明 |
|------|------|
| 首批书籍选择 | 基于 Phase 1 TTS 使用数据，选最受欢迎的书优先 |
| 轨道分配 | 有 LibriVox 高质量录音 → A，无 → B |
| 管道执行 | 各轨道并行生产 |
| 人工审核 | 确认音质、章节对齐 |
| R2 存储上传 | 音频文件 CDN 分发 |
| 数据库写入 | Audiobook + AudiobookChapter 记录 |

**交付物**：首批 50 本精品有声书数据入库

---

### Phase 4: 播放体验

**目标**：用户在 App 中播放精品有声书，体验升级

| 任务 | 说明 |
|------|------|
| 音频优先策略 | 有精品音频 → 播放精品；无 → 回退到 TTS |
| 播放器升级 | 进度记忆、倍速、定时关闭 |
| 音频下载 | 离线收听支持 |
| 用户进度同步 | 阅读进度 ↔ 收听进度关联 |

```
用户点击"听书"
     ↓
该书有精品音频？
  ├── 是 → 播放精品音频（真人重制 or AI 原创）
  └── 否 → 回退客户端 TTS
```

**交付物**：用户能听到精品有声书，且无精品时自动 TTS 兜底

---

### Phase 5: 章节级音频+文本同步（终极体验）

**目标**：音频播放时同步高亮文本，翻译联动

| 任务 | 说明 |
|------|------|
| Forced alignment | 使用 WhisperX/Aeneas 生成段落时间戳 |
| timestamps 数据生成 | 填充 AudiobookChapter.timestamps |
| 客户端同步播放 | 音频播放时高亮对应段落 |
| 翻译联动 | 播放到某段时显示该段翻译 |

**交付物**：文本+音频+翻译 三位一体的沉浸式体验

## 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| TTS 用户反馈差 | Phase 1 数据失真 | 明确标注"AI 朗读"，管理预期 |
| SE 书在 LibriVox 覆盖率低 | 轨道 A 规模小 | 轨道 B 补位，覆盖率不是瓶颈 |
| 章节结构差异大（轨道 A） | 对齐困难 | 转入轨道 B 原创制作 |
| AI TTS 成本高（轨道 B） | 预算超支 | 先用免费/低价 TTS，热门书才用高质量 TTS |
| 音频存储成本 | R2 费用 | 先小批量验证成本模型 |
| 版权风险 | 法律问题 | LibriVox（CC/PD）、SE 文本（PD）、AI TTS（检查服务条款） |

## 研发节奏

```
Phase 1 (客户端 TTS)
  ├── 纯客户端实现，API 无改动
  ├── 上线收集数据
  └── 输出：听书需求验证 + 热门书排行

Phase 2 (Audio Lab 双轨制管道)
  ├── 轨道 A: LibriVox 目录抓取 + 匹配摸底

  └── 输出：双轨制生产管道 + 覆盖率报告
        │
        └── 决策点：轨道 A 覆盖率如何？
              ├── 高 → 轨道 A 为主，B 补充
              └── 低 → 轨道 B 为主，A 补充

Phase 3 (精品制作与导入)
  ├── 首批 50 本（基于 TTS 使用数据选书）

Phase 4 (播放体验)
  ├── 精品优先 + TTS 兜底
  ├── 播放器功能完善
  └── 输出：用户可听精品有声书

Phase 5 (深度同步)
  ├── Forced alignment 时间戳
  ├── 同步播放 + 翻译联动
  └── 输出：三位一体沉浸式体验
```

### 3.4 数据库迁移规则

| 迁移类型 | 允许 | 说明 |
|----------|------|------|

### 5.2 API 升级策略

| 升级类型 | 策略 | 影响 |
|----------|------|------|
| **向后兼容升级** | 直接在 v1 添加 | 不影响现有客户端 |

| **破坏性变更** | 创建 v2 版本 | v1 保持兼容运行 |

## 七、风险与回滚

### 7.1 风险清单

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| API 接口变更 | 客户端崩溃 | API 变更必须向后兼容 |
| 数据不完整 | 内容缺失 | 发布前完整验证 |
| 性能问题 | 用户体验差 | Staging 性能测试 |
| 审核被拒 | 发布延迟 | 遵循 Apple 指南 |

### 7.2 回滚策略

```
┌─────────────────────────────────────────────────────────────────┐
│                        回滚策略                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   API 回滚                                                       │
│   ├── 保留上一个稳定版本的 Docker 镜像                           │
│   ├── fly deploy --image <previous-image>                        │
│   └── 回滚时间 < 5 分钟                                          │
│                                                                   │
│   数据回滚                                                       │
│   ├── 保留数据库快照 (fly postgres backup)                       │
│   ├── R2 版本控制 (文件历史)                                     │
│   └── 回滚时间 < 30 分钟                                         │
│                                                                   │
│   iOS 回滚                                                       │
│   ├── 无法直接回滚已发布版本                                     │
│   ├── 紧急修复版本 (expedited review)                            │
│   └── 服务端降级处理                                             │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```
